# Monitoring

Last verified against Cloudflare documentation: 2026-07-15.

## What to monitor and where

| Signal | Source | Alert condition (suggested) |
|---|---|---|
| Edge Worker error rate | Workers Analytics (dashboard or GraphQL Analytics API) | 5xx rate > baseline for 5 min |
| Edge Worker CPU/latency | Workers Analytics | p99 latency regression |
| `GET /healthz` (edge-only) | External uptime monitor hitting `https://api.amelu.org/healthz` | non-200 for 2+ consecutive checks |
| `GET /healthz/upstream` (edge+origin) | External uptime monitor | non-200 for 2+ consecutive checks - distinguishes "edge down" from "origin/tunnel down" when compared against the check above |
| Tunnel connector state | Zero Trust dashboard, or `cloudflared`'s own Prometheus metrics endpoint if enabled | Any replica disconnected > 5 min |
| Queue backlog depth | Queues dashboard / GraphQL Analytics | Backlog not draining over 30+ min |
| DLQ message count | Queues dashboard | Any message present (should normally be zero or near-zero) |
| Origin (Go) process health | Existing process supervisor/host monitoring, unchanged by this migration | unchanged from pre-migration baseline |

Reference: https://developers.cloudflare.com/analytics/graphql-api/ and
https://developers.cloudflare.com/queues/observability/metrics/

## Distinguishing edge vs. origin outages

The two health endpoints (`EDGE_WORKER.md`) are the primary tool: if
`/healthz` is 200 but `/healthz/upstream` is 502, the problem is the
Tunnel/origin, not the Worker - route the on-call accordingly. If `/healthz`
itself is failing, it's a Worker/Cloudflare-platform issue.

## Request id correlation

Every response carries `X-Request-Id` (generated by the Worker if the
client didn't supply one). Include it when filing/searching an incident so
Worker logs (`wrangler tail`) and origin logs can be correlated for the same
request.

## Structured logging

Worker logs are JSON lines (`console.log(JSON.stringify({...}))` throughout
`cloudflare/edge/src` and `cloudflare/queues/domain-verification/src`) - not
because Cloudflare requires it, but so `wrangler tail`'s output or any log
drain configured later can be filtered/parsed without adding a separate
logging library dependency.

## Setting up a log drain (optional, not configured by this migration)

Workers Logpush can stream Worker execution logs to an external
destination (R2, S3, a SIEM, etc.) for longer retention than the dashboard's
live tail provides. Not configured here - a dashboard/API action with cost
implications (see `COSTS.md`), left as an operator decision once real
traffic volume is known.

Reference: https://developers.cloudflare.com/workers/observability/logs/logpush/

## Alerting

Cloudflare Notifications (dashboard -> Notifications) can alert on Worker
error rate thresholds, Tunnel health, and Queue backlog directly without
external tooling - not configured by this migration since thresholds should
be tuned against real traffic, not guessed at build time.

Reference: https://developers.cloudflare.com/notifications/

## Synthetic checks

Recommended, not implemented here: an external uptime monitor (outside
Cloudflare, so it independently verifies Cloudflare itself is reachable)
hitting both health endpoints on a 1-2 minute interval, plus one
synthetic end-to-end check (log in, list domains) against a dedicated test
account, to catch a bug in the proxy path (e.g. cookie handling) that a
plain health check wouldn't.
